# VITRIOL Optimization Findings — 2026-05-19

## Overview

Benchmarking and optimizing Qwen3.6-35B-A3B on GTX 1070 Ti (8 GB VRAM, Pascal CC 6.1)
with VITRIOL expert offloading via PCIe Gen3 x16.

System: 15 GB RAM, NVMe SSD, 4 CPU threads.

---

## 1. PCIe x16 Restoration — The Dominant Gain

### Problem
GTX 960 (CC 5.2, no kernel images for VITRIOL) was installed in the second PCIe slot.
This halved the primary slot from **x16 → x8**, reducing bandwidth from **15.76 GB/s → 7.88 GB/s**.

### Diagnosis
Gen speed was stuck at ~5.7 tok/s regardless of LRU cache size, thread count, or context length.
PCIe bandwidth bottleneck was suspected because VITRIOL streams experts over PCIe on every token.

### Fix
Physically removed GTX 960. Primary slot restored to x16.

### Result
Gen speed jumped from **5.7 → 9.1 tok/s** — a **+60% improvement**.

### Why this matters
VITRIOL loads 8 experts per MoE layer × ~5 layers per token = ~40 expert tensors over PCIe per forward pass.
At 8-bit quant (2.2-2.6 bpw), that's ~40-80 MB/token across the bus. PCIe x8 becomes the bottleneck.
x16 roughly doubles available bandwidth.

## 2. MTP Speculative Decoding — Incremental Gain

### Setup
- Model: `Qwen3.6-35B-A3B-UD-IQ2_M.gguf` (Unsloth MTP repo, IQ2_M quant 2.6 bpw)
- Has `nextn_predict_layers` (MTP head: 1 layer, 256 experts)
- Baseline for comparison: `Qwen3.6-35B-A3B-UD-Q2_K_XL.gguf` (Q2_K_XL quant ~2.2-2.4 bpw, no MTP)

### Integration
VITRIOL's expert tensor pattern matching (`'exps'` glob) catches `blk.40.ffn_*_exps` from the MTP head
context automatically. No separate integration needed — both main trunk and draft model load through the
same VITRIOL buffer.

### Result at N=2
| Metric | Without MTP | With MTP (N=2) | Improvement |
|--------|------------|----------------|-------------|
| Gen (tok/s) | 9.10 | 10.96 | +20% |
| Eval (tok/s) | 58.2 | 83.0 | +43% |
| Draft acceptance | — | 50% | 1/2 per cycle |

### MTP Draft-N-Max Sweep (2026-05-19)

Tested N = 2, 3, 4, 6, 8, 12 on IQ2_M model at 256K context, Q4_0 KV, frozen prompt:

| DraftN | N50 gen | N50 p.eval | N50 accept | N100 gen | N100 p.eval | N100 accept |
|--------|---------|------------|------------|----------|-------------|-------------|
| 2      | **10.96** | 82.97    | 24/48 (50.0%) | **10.68** | 83.29    | 49/98 (50.0%) |
| 3      | 10.07    | 82.62      | 24/71 (33.8%) | 9.93     | 83.14      | 49/146 (33.5%) |
| 4      | 9.91     | 83.13      | 24/94 (25.5%) | 9.77     | 83.12      | 49/194 (25.2%) |
| 6      | 9.34     | 83.01      | 24/138 (17.3%) | 8.43    | 82.74      | 49/288 (17.0%) |
| 8      | 7.36     | 73.29      | 24/180 (13.3%) | 7.29    | 67.68      | 49/380 (12.8%) |
| 12     | 6.70     | 68.55      | 24/258 (9.3%) | 6.59     | 71.62      | 49/558 (8.7%) |

### Key finding: N=2 is optimal

The acceptance rate follows exactly `1/N` — the model accepts **exactly 1 token** per draft cycle
regardless of N. All draft tokens beyond the first are wasted compute:

```
N=2:  24/48  = 50.0%  (1.00 accepted per cycle)
N=3:  24/71  = 33.8%  (1.01 accepted per cycle)
N=4:  24/94  = 25.5%  (1.02 accepted per cycle)
N=6:  24/138 = 17.3%  (1.04 accepted per cycle)
N=12: 24/258 =  9.3%  (1.12 accepted per cycle)
```

Since each additional draft token requires a full MTP head forward pass (fetching 8 experts
over PCIe + compute), higher N just adds overhead without benefit.

**N=2 is the sweet spot** on this setup. N=3 is within ~8% but not worth the extra compute.

Prompt eval is unchanged (~83 tok/s) until N=8+, where the draft model overhead during
prompt processing starts to matter.

### VRAM with MTP at 256K
- Model weights (GPU): 1337 MiB (main) + 302 MiB (MTP head)
- VITRIOL buffer (RAM): ~10040 MiB
- KV cache (host, Q4_0): ~1406 MiB
- Total system RAM: ~13.5 GiB
- Total VRAM: ~3235 MiB (headroom: ~4877 MiB on 8 GB)

### Caveat
Different quants between baseline (Q2_K_XL) and MTP (IQ2_M). The baseline model *has no MTP layers*,
so this is not an apples-to-apples comparison of the same model with/without MTP — it's a
Q2_K_XL vs IQ2_M model comparison where the IQ2_M happens to have MTP layers. The +20% gain
is net of any quality vs speed tradeoff from the different quant.

## 3. Floundering Log — Time Lost to Wrong Settings

### Episode 1: Silent Server Death (~2 hours)

**Attempt**: Run `llama-server` in background with `nohup` and all VITRIOL env vars.

**Observed**: Process started, logged a few lines (`llama_model_loader`, `build info`), then
silently died a few seconds later. No crash message. `dmesg` showed no OOM killer.

**Debug trail**:
1. Checked VRAM — only 3-4 GB used, well within 8 GB limit.
2. Checked RAM with `free -h` — ~12-13 GB used, looking tight.
3. Ran `llama-server` **foreground** — worked fine! Only background mode failed.
4. Hypothesis: CUDA context requires a controlling terminal.
5. Tried `setsid`, `screen`, `tmux` — all same death pattern.
6. Logged every env var — noticed `VITRIOL_KV_QUANT=q4_0` was set in env, but
   `--cache-type-k q4_0 --cache-type-v q4_0` was **not** on the command line.

**Root cause**: The env var `VITRIOL_KV_QUANT` controls how VITRIOL *stores* KV in its
buffer, but `llama-server` needs explicit `--cache-type-k/v` flags to quantize its host
context allocation:

- F16 KV cache allocation at 256K context: **~5000 MiB**
- Q4_0 KV cache allocation: **~1406 MiB**
- VITRIOL buffer: **~10040 MiB**
- F16 math: 5000 + 10040 = 15040 MiB ≈ 14.7 GiB
- With compute buffers + OS overhead: **exceeds 15 GB RAM → silent death**

**Fix**: Always pass `--cache-type-k q4_0 --cache-type-v q4_0` to `llama-server`.
This is now wired into the `vitriol` script automatically from the `[kv] quant_mode` config.

**Lesson**: The env var and the CLI flag are **independent**. Env var alone is insufficient.
Both must be set. The vitriol script now handles this.

### Episode 2: Nohup vs Foreground — Still Not Fully Understood

Even after fixing the KV flags, `env CUDA_VISIBLE_DEVICES=0 nohup llama-server [...] &`
would sometimes die while the foreground version (`env CUDA_VISIBLE_DEVICES=0 ./llama-server [...]`)
worked fine.

Observations:
- Not consistent — sometimes nohup worked, sometimes not
- `vitriol serve --detach` worked reliably (uses the same nohup internally but with
  all env vars exported before launch)
- Possible cause: CUDA driver initializes GPU context lazily on first kernel launch.
  When the process is backgrounded and its stdout/stderr are redirected, CUDA may fail
  to bind to the GPU context if the terminal session disconnects.
- Workaround: Use `vitriol serve --detach` (or `vitriol serve` for foreground).

### Episode 3: `llama-cli` Build Broken (ongoing)

The `llama.cpp/tools/cli/cli.cpp` file in the MTP branch references:
- `reasoning_budget` (should be `reasoning_budget_tokens` in the API)
- `reasoning_budget_message` (missing field)
- `llama_memory_breakdown_print` (function not exported from `libllama.so.0`)

This blocks `llama-cli` rebuild. `llama-server` builds fine. Workaround: use the server
binary exclusively (VITRIOL only needs the server anyway).

## 4. Lessons Learned Checklist

These flags/settings are **mandatory** for VITRIOL to work correctly at 256K context:

| Setting | Why | How |
|---------|-----|-----|
| `--cache-type-k q4_0` | Reduces host KV from 5000→1406 MiB | Config `[kv] quant_mode = q4_0` |
| `--cache-type-v q4_0` | Same as above | Config `[kv] quant_mode = q4_0` |
| `--no-mmap` | Required for VITRIOL DMA buffer | Hardcoded in vitriol script |
| `-fa on` | Flash attention required for V cache quant | Hardcoded in vitriol script |
| `vitriol-dma` engine | Enables expert intercept | Config `[engine] mode = vitriol-dma` |
| `stream` mode | Expert offloading via page-locked RAM | Config `[vitriol] mode = stream` |
| `frozen_prompt = on` | Avoids re-prefix on each request | Config `[kv] frozen_prompt = on` |
| `exclude_secondary = true` | GTX 960 (CC 5.2) lacks kernel images | Config `[gpu] exclude_secondary = true` |

## 5. Benchmark Procedure

Current benchmark method:
1. Start `llama-server` via `vitriol serve --detach` with test config
2. Wait for `/health` endpoint to respond
3. Warm up: `max_tokens=10` curl request to `/v1/chat/completions`
4. Run at `max_tokens=50` and `max_tokens=100` with sequential log line counting
5. Read gen speed (tok/s) from `eval time =` line in server log
6. Read prompt eval speed (tok/s) from `prompt eval time =` line
7. For MTP: read draft acceptance from `draft acceptance rate` log line
8. Kill server (`vitriol stop`), repeat for each config change
9. Automated sweep script: `scripts/benchmark_mtp_sweep.sh`

## 8. Compute-Bound Optimization: Fewer Experts + Output Cache

### Background
LRU VRAM pool cannot improve generation throughput (see §5). PCIe is not the bottleneck
(0.18 ms per token vs 91 ms total). The bottleneck is GPU compute: dequantizing + mat-vec
300M parameters/token on GTX 1070 Ti (6.1 TFLOPS).

Two complementary optimizations were implemented to reduce compute:

### Option 1: Fewer Active Experts

**What**: Runtime override of `n_expert_used` from model default (8) to 4 or 6.

**Implementation**:
- `llama_cparams::n_expert_used_override` field (default 0 = model default)
- `llm_graph_context` initialization checks override at `llama-graph.cpp:930`
- `VITRIOL_EXPERT_COUNT` env var (set via TUI option 5 in Model Settings)

**Expected effect**: 25-50% compute reduction. Quality impact unknown for Qwen3.5/3.6.
Router weights concentrate on 2-3 experts, so bottom experts contribute minimally.

### Option 2: Approximate Expert Output Cache

**What**: Cache expert FFN outputs between consecutive tokens. If expert E is in top-K
for both token N and N+1, reuse output from N instead of recomputing.

**Implementation**:
- Per-layer, per-expert VRAM cache (128 layers × 256 experts × 2048 floats = ~256 MB)
- `VITRIOL_OUTPUT_CACHE=1` env var (TUI option 6 in Model Settings, requires stream mode)
- Forces sorted path during single-token generation when active (to reach cache hooks)
- Cache lookup before each per-expert `ggml_cuda_mul_mat` call in sorted path
- On hit: D2D copy cached output, skip matmul
- On miss: compute normally, store output in cache for next token

**Important caveat**: This is APPROXIMATE — expert outputs are token-dependent.
Cached outputs from token N produce stale results for token N+1.
Accepted tradeoff for speed. Quality impact to be measured.

**Integration with predictor**: Cache uses the same `get_layer_index` infrastructure
as the cross-layer/temporal predictor. Cache does NOT invalidate between tokens
(that's the whole point — persistence enables hits).

### Combined Effect
With top-4 and 2-3 cached: only 1-2 new experts computed per layer.
Could reach ~2× compute reduction during generation.
