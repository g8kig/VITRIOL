# Optimization Plan: Fewer Active Experts + Expert Output Caching

## Goal
Reduce GPU compute during generation on GTX 1070 Ti by:
1. **Fewer active experts**: Override top-K from 8 to 4 or 6 (25-50% less compute)
2. **Expert output caching**: Skip recomputation of experts carried over from the previous token

## Bottleneck Context
- Throughput is compute-bound (not PCIe-bound) — per-token compute for 8 experts × 300M params dominates
- LRU VRAM pool (which saves PCIe traffic) cannot help during generation
- These optimizations directly reduce compute, which is the actual bottleneck

## Option 1: Fewer Active Experts

**What**: Runtime override `n_expert_used` from model default (8) to 4 or 6.

**Changes**:
- `src/llama-cparams.h`: Add `int32_t n_expert_used_override = 0`
- `src/llama-graph.cpp:930`: Use `cparams.n_expert_used_override` if non-zero
- `scripts/vitriol`: Add `expert_count` config key + TUI option

**Quality concern**: May degrade quality. Router weights often concentrate on 2-3 experts; the bottom 4-5 contribute minimally.

## Option 2: Expert Output Caching (Fate-style)

**What**: During generation, experts used by token N that are also selected for token N+1 have their output reused instead of recomputed. Only new experts execute the matmul.

**Changes**:
- `vitriol-cuda-integration.h/cpp`: Per-layer cache of expert outputs from the previous token
- `ggml-cuda.cu` sorted path: Before each per-expert `ggml_cuda_mul_mat`, check cache; on hit, D2D copy cached output to dst_slice; on miss, compute normally and store in cache
- `ggml-cuda.cu` MMV path: Force sorted path during single-token generation when cache is active (so cache hooks are reached)

**Expected gain**: 20-40% compute reduction from skipping 2-4 expert recomputations per layer.

## Combined Effect
With top-4 and 2-3 cached: only 1-2 new experts computed per layer. Could reach ~2× speedup on generation.

## Interaction
- Options 1 and 2 are independent and complementary
- Option 2 alone may already provide significant gains since expert reuse between adjacent tokens is ~30-50%
- Combined: cache reduces the compute cost of using a higher `n_expert_used` value (e.g., use top-12 but rely on cache hits)
